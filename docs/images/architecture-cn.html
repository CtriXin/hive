Hive v2.1.0 主线

Hive 架构图

当前 reality：CLI、MCP、以及本地 Web 共用同一套 run-time control loop。

入口面 入口不同，但底层 runtime 是同一套

hive CLI
从 terminal 发起 run、恢复执行、观察状态、提交 steering、查看 compact/restore、以及打开 Web。
run / resume status / watch compact / shell / web
MCP tools
给宿主提供结构化入口，用于 planning、execution、reporting、run 查询以及 steering。
plan_tasks run_goal / run_status submit_steering
hive web
本地 browser decision surface，聚焦 stop reason、next action、active runs，以及模型策略中心。
decision-first UI 模型策略中心

核心控制循环 一次 run 在 Hive 里真实发生的事情

Run 启动 + 状态机
创建 `spec.json` / `state.json`，管理 phases、terminal states、retry，以及 loop progress。
spec / state / progress
→
Planner + driver loop
负责 plan、resume、repair vs replan 决策，并维持整个 operator loop 的一致性。
plan / execute / repair / replan
→
能力路由 + provider 韧性
负责 effective model 选择、fallback、bounded retry、cooldown，以及 route health。
routing / fallback / retry
→
隔离 git worktree workers
每个 task 在自己的 worktree 里执行，这样可以并行、可 review、也更安全地 merge。
parallel task execution
→
Review 级联 + verification + merge
通过 cross-review、authority、build/test verification，以及 progressive merge 来真正收口。
review / verify / merge
分层策略 + steering
现在不是“全局拍脑袋配置”。Hive 会跨层解析 effective policy，并对正在运行的 run 保持 safe-point 生效语义。
Run > Project > Global > Default safe-point override
Discuss / advisory / memory
worker 不确定性、advisory 信号、以及 recall 上下文，会真实进入 control loop，而不是留在边上。
人工 steering
暂停、继续、请求 replan 等操作会持久化，并在 safe point 被消费，而不是临时 UI 按钮。

运行产物与观察面 先有持久化 run truth，再有 CLI / Web 呈现

.ai/runs/<run-id>
每个 run 都会落盘 plan、result、loop progress、worker status、score history、steering state、provider health 和 model overrides。
plan / result worker-status / score-history model-overrides / provider-health
CLI 观察面
status、watch、compact、restore、shell、workers、runs 都读取持久化 run artifacts。
Web 观察面
browser 页面主要回答 verdict、blocker、next action、active project/run，以及模型策略中心。
这是本地 browser Web，不是 hosted SaaS dashboard
重点是 operator clarity 与可复现 artifacts

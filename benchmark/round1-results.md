# Hive 国产模型多维度综合测评

> 测评时间：2026-03-27
> 测评轮次：Round 1（初次）
> 评测方式：统一题目 → 各模型独立作答 → Claude 交叉打分

## 一、测评题目

所有模型使用完全相同的 prompt，包含 4 个维度的任务。

### TASK1: Coding — 泛型 LRU Cache

```
用 TypeScript 实现一个泛型 LRU Cache（Least Recently Used），要求：
- 支持任意 key/value 类型
- get() 和 put() 时间复杂度 O(1)
- 容量满时自动淘汰最久未使用的项
- 提供 delete(key) 和 clear() 方法
- 提供完整的 TypeScript 类型约束
- 包含至少 3 个边界用例的测试代码
```

**考察点：**
- 是否使用双向链表 + HashMap 实现真正的 O(1)（而非依赖 Map 插入顺序的"伪 O(1)"）
- 泛型约束是否完整
- 边界测试覆盖度（容量为 1、频繁访问改变顺序、delete/clear、复杂类型 key）
- 代码质量（命名、注释、结构）

### TASK2: Reasoning — 工厂产能推理

```
一个工厂有 3 条生产线 A/B/C，每天总产能 1000 件。
- A 线效率是 B 线的 1.5 倍
- B 线效率是 C 线的 2 倍
- A 线每天维护停机 2 小时，B 线停机 1 小时，C 线不停机
- 其余时间（每天按 8 小时计）三条线同时满载运行

问题：
1. 每条生产线每小时的产能各是多少？
2. 一天实际总产量是多少？
3. 如果取消 A 线的维护停机，日产量增加多少百分比？

请展示完整的推理过程。
```

**考察点：**
- 变量定义与关系推导是否清晰
- 对"每天总产能 1000 件"的理解（标准答案：实际运行时间产出之和 = 1000）
- 计算过程是否完整、可验证
- 最终答案是否正确

**标准答案：**
- 设 C 线每小时产能为 x，则 B=2x，A=3x
- 6(3x) + 7(2x) + 8(x) = 1000 → 40x = 1000 → x = 25
- A=75 件/h，B=50 件/h，C=25 件/h
- 日产量 = 450 + 350 + 200 = 1000 件
- 取消 A 停机后 = 600 + 350 + 200 = 1150，增加 15%

**存在歧义的第二种解读：**
- "总产能 1000" = 理论满载 8h 产能 → 48x = 1000 → x = 125/6
- 日产量 = 40(125/6) ≈ 833 件
- 百分比仍为 15%（6x/40x = 15%）
- 此解读合理但非最优，扣 1-2 分

### TASK3: Translation — 中英日三语技术翻译

```
将以下技术段落翻译为英文，然后翻译为日文，最后再将英文版翻译回中文。保持技术术语的一致性。

原文：
"在微服务架构中，服务网格（Service Mesh）通过 sidecar 代理实现流量管理、服务发现和负载均衡。
Istio 作为最流行的服务网格实现，其数据平面由 Envoy 代理组成，控制平面则负责配置分发和证书管理。
蓝绿部署和金丝雀发布是服务网格中常见的发布策略，配合流量镜像（Traffic Mirroring）可以实现零风险的版本切换。"
```

**考察点：**
- 英文翻译准确性（技术术语、句式流畅度）
- 日文翻译准确性（片假名使用、语法正确性）
- 回译还原度（与原文的差异越小越好）
- 术语一致性（Service Mesh / sidecar / Envoy / canary release 等跨语言一致）

**扣分项：**
- 日文混入中文字词（如"服務メッシュ"应为"サービスメッシュ"）
- "ブル-green展開"等明显错误
- 漏译或误译关键术语

### TASK4: Analysis — 分布式 IM 系统架构设计

```
你正在为一个日活 1000 万的社交平台设计即时消息系统。当前架构是单体应用 + Redis Pub/Sub，已遇到性能瓶颈。

请给出一个分布式消息架构方案，需要涵盖：
1. 整体架构图（用 ASCII 或文字描述）
2. 消息投递的可靠性保证策略（至少覆盖：离线消息、消息不丢失、消息不重复）
3. 与现有系统的平滑迁移方案（要求零停机）
4. 性能指标估算（QPS、延迟、存储）
5. 方案的潜在风险和应对措施
```

**考察点：**
- 架构完整性（接入层 → 消息总线 → 处理层 → 存储层）
- 可靠性三大问题是否都有具体方案（非空话）
- 迁移方案是否可落地（阶段划分、回滚策略）
- 性能估算是否有数据支撑（而非拍脑袋）
- 风险识别深度（是否覆盖常见陷阱：扇出风暴、消息乱序、冷启动等）

## 二、评分标准

每维度 1-10 分，总分 40 分。

### Coding 评分细则

| 分数 | 标准 |
|------|------|
| 9-10 | 双向链表 + HashMap 经典实现，泛型完整，>=4 组边界测试，代码整洁 |
| 7-8 | Map 利用插入顺序实现（有效但非经典），基本测试覆盖，功能正确 |
| 5-6 | Map 实现+测试但不够完善，或链表实现有小 bug |
| 3-4 | 算法正确性有问题（如 O(n) indexOf 破坏 O(1) 要求） |
| 1-2 | 功能缺失或严重 bug |

### Reasoning 评分细则

| 分数 | 标准 |
|------|------|
| 10 | 答案正确 + 推理链清晰 + 有验证步骤 |
| 8-9 | 答案正确但对题意解读有可争议之处（如第二种解读） |
| 6-7 | 答案正确但推理过程混乱或有计算错误后自纠 |
| 4-5 | 最终答案错误但有合理推导 |

### Translation 评分细则

| 分数 | 标准 |
|------|------|
| 9-10 | 三语翻译准确流畅，术语一致，回译高度还原 |

| 5-6 | 有明显翻译错误但整体可理解 |
| 3-4 | 日文翻译混入大量中文或严重误译 |

### Analysis 评分细则

| 分数 | 标准 |
|------|------|
| 9-10 | 5 项全覆盖且有深度，架构图清晰，量化数据合理，风险识别深入 |
| 7-8 | 基本覆盖但某几项偏浅，或有明显遗漏 |
| 5-6 | 覆盖不全或缺乏数据支撑 |

## 四、关键发现

### 模型家族特征

| 家族 | 特征 |
|------|------|
| **GLM（智谱）** | 均衡稳定，Coding 功底扎实，Analysis 适中。glm-5 最快（85s），glm-4.7 综合最优 |
| **Kimi（月之暗面）** | kimi-for-coding 综合第一，Analysis 深度突出。kimi-k2.5 Coding 用纯 Map 略逊 |
| **MiniMax** | M2.7 Analysis 最详尽但最慢（159s），M2.5 翻译出硬伤（日文混中文） |
| **Qwen（阿里）** | 差距最大。qwen3.5-plus Coding 扎实但 Reasoning 偏差；qwen3-max Coding 出 O(n) 硬伤；qwen3-coder-plus 最快但最浅 |

### 速度 vs 质量相关性

| 速度排名 | 模型 | 时间 | 总分排名 |
|---------|------|------|---------|
| 1 | qwen3-coder-plus | 34s | 9 (31.5) |
| 2 | glm-5 | 85s | 3 (36.0) |
| 3 | kimi-k2.5 | 94s | 5 (35.0) |
| 4 | qwen3-max | 93s | 10 (27.0) |
| ... | qwen3.5-plus | 166s | 7 (34.5) |
| 最慢 | MiniMax-M2.7 | 159s | 4 (35.5) |

**结论：速度和质量无强相关性。qwen3-coder-plus 最快但质量倒数第二；glm-5 又快又好。**

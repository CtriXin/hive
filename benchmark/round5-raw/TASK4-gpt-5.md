# TASK4: 分布式系统故障诊断分析

## 系统架构

```
Client → API Gateway → Services + Redis Cluster + Kafka
```

## 一、完整故障链分析

### 触发阶段：Redis Failover
- Redis Cluster 发生主从切换（failover），持续时间约 6 秒
- 切换期间，主节点不可用，客户端请求 Redis 超时或返回 `MOVED`/`ASK` 重定向

### 放大阶段：重试风暴
- 应用层对 Redis 异常配置了固定间隔重试，重试总窗口长达 20 秒
- 每次重试都占用新的连接或阻塞现有连接
- 由于 Redis 在 6 秒后已恢复，但应用仍在对旧连接/超时请求进行无效重试

### 崩溃阶段：连接池耗尽
- 大量等待 Redis 响应的线程/协程长时间占用数据库/HTTP/RPC 连接池中的连接
- 新到达的下单请求无法获取连接，在 API Gateway 或 Service 层超时
- 级联效应：上游 Gateway 等待下游响应超时，返回 500

### 表现阶段：Kafka 侧影响（潜在）
- 若下单流程包含异步发 Kafka 做后续处理，连接池耗尽可能导致消息延迟堆积
- 恢复后可能出现短暂的消费延迟补偿

---

## 二、为什么是"间歇性"（2-3%）

1. **Redis failover 本身是小概率事件**：仅在节点故障、网络分区、手动切换时触发，并非持续存在
2. **流量窗口与故障窗口的交集概率低**：只有在 failover 发生的 6 秒内且有并发下单请求时才会触发
3. **连接池耗尽需要并发条件**：单机或低并发时，连接释放速度足以应对；高并发时才出现饱和
4. **部分请求可能命中从节点或缓存命中其他键槽**，不会全部失败
5. **2-3% 的 500 率恰好对应 failover 期间 QPS 占日峰值的比例**，说明故障影响范围与请求分布成正比

## 三、修复建议（≥5条）

### 1. 缩短 Redis 重试策略（高优先级）
- 将 20 秒的重试窗口改为 **指数退避 + 快速失败（fail-fast）**
- 建议最大重试时间 ≤ 3 秒，与 Redis failover 窗口匹配

### 2. 配置合理的连接池大小与超时
- 设置连接池最大等待时间（如 1 秒），超时立即拒绝，避免线程长期阻塞
- 启用连接池健康检查，自动剔除不可用连接

### 3. 增加 Redis 集群客户端的缓冲与熔断
- 使用支持 cluster-aware 的客户端（如 Lettuce、Jedis Cluster、go-redis）
- 配置 **熔断器（Circuit Breaker）**：Redis 连续失败率达到阈值后，短时间内直接降级，不再访问 Redis

### 4. 引入本地缓存降级
- 对热点数据（如商品库存、用户信息）增加 **Caffeine/Guava Cache** 本地缓存
- Redis 不可用时，读本地缓存并Allow-Stale，保证下单核心链路可用

### 5. 优化 API Gateway 的超时与限流策略
- 缩短 Gateway → Service 的超时时间（如 3 秒），避免长尾请求阻塞
- 启用 **自适应限流** 或 **令牌桶限流**，在下游异常时自动削峰

### 6. 异步化下单非核心路径
- 将积分、优惠券、日志写入等操作异步化（发 Kafka）
- 保证核心链路（扣库存、创建订单、支付）尽量同步且最短

### 7. Redis 部署优化
- 缩短 `cluster-node-timeout` 配置，加速故障检测与自动切换
- 增加 Redis 从节点或使用 **Proxy 层（如 Codis、Twemproxy）** 屏蔽 cluster 拓扑变化对应用的影响

## 四、架构层面预防措施

| 层级 | 措施 |
|------|------|
| **客户端** | 合理超时、指数退避、快速失败、熔断降级 |
| **服务层** | 本地缓存、异步化、核心链路隔离、连接池监控 |
| **网关层** | 超时熔断、自适应限流、错误码规范化（避免统统返回 500） |
| **Redis 层** | 多可用区部署、自动故障转移优化、Proxy 屏蔽拓扑变化 |
| **可观测性** | 全链路追踪（TraceID）、Redis 连接池实时监控、P99/P99.9 告警 |
| **混沌工程** | 定期注入 Redis failover，验证系统韧性与恢复能力 |

---

## 五、优先级排序

| 优先级 | 措施 | 原因 |
|--------|------|------|
| P0 | 缩短 Redis 重试 + 连接池超时控制 | 直接消除根因，成本低，见效快 |
| P0 | 引入熔断器 + 快速失败 | 阻断级联故障蔓延 |
| P1 | 本地缓存降级 | 提升故障期间的可用性 |
| P1 | Gateway 超时与限流优化 | 保护上游，防止雪崩 |
| P2 | 异步化非核心链路 | 缩短核心链路，提升整体吞吐 |
| P2 | Redis 部署与参数调优 | 减少 failover 频率与持续时间 |
| P3 | 全链路追踪与混沌工程 | 长期体系建设，预防未知风险 |

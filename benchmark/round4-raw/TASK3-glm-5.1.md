# 限时秒杀功能模块 — 任务规划

## 1. 子任务拆分

| ID | 描述 | 复杂度 | 预估文件 | 依赖 |
|----|------|--------|----------|------|
| T1 | 秒杀商品管理 API（CRUD + 库存预扣接口设计） | 中 | 4-5 文件 | 无 |
| T2 | 高并发库存扣减服务（Redis + Lua 脚本防超卖） | 高 | 3-4 文件 | T1（库存预扣接口契约） |
| T3 | 防刷策略中间件（IP 限流 + 用户频率控制 + 验证码集成） | 中 | 3-4 文件 | 无 |
| T4 | 秒杀订单异步处理队列（消息队列生产者/消费者） | 高 | 4-5 文件 | T2（扣减成功后触发下单） |
| T5 | 秒杀倒计时前端组件（服务端时间同步 + 毫秒倒计时） | 中 | 3-4 文件 | T1（获取秒杀活动时间） |
| T6 | 秒杀结果 WebSocket 推送服务 | 中 | 3-4 文件 | T4（订单结果作为推送源） |
| T7 | 监控告警面板（Prometheus 指标 + Grafana Dashboard JSON） | 低 | 2-3 文件 | T2, T4（埋点依赖核心服务） |

---

## 2. 并行执行组

```
Group 0 (无依赖，立即并行):  T1, T3
Group 1 (依赖 T1):          T2, T5
Group 2 (依赖 T2):          T4
Group 3 (依赖 T4):          T6
Group 4 (依赖 T2+T4):       T7
```

```
T1 ──┬──→ T2 ──→ T4 ──→ T6
     │              │
     └──→ T5        └──→ T7 ←── T2
T3 ─────────────────────────→ (独立，全程并行)
```

关键路径: **T1 → T2 → T4 → T6**（4 个串行节点，其中 T3 全程并行）

## 3. 模型分配方案

| 任务 | 分配模型 | 理由 |
|------|----------|------|
| T1 商品管理 API | **kimi-k2.5** | CRUD + 接口契约设计，需要稳定可靠不出错，k2.5 均衡无硬伤，~100s 性价比高 |
| T2 库存扣减 Lua 脚本 | **kimi-for-coding** | 核心难点：Lua 原子操作 + Redis 集群 + 防超卖竞态。需要深度分析能力，综合最强处理最复杂任务 |
| T3 防刷策略中间件 | **qwen3.5-plus** | 限流算法（令牌桶/滑窗）+ 验证码集成，编码扎实即可，~140s 适中 |
| T4 异步订单队列 | **kimi-for-coding** | 消息可靠性、幂等消费、死信队列、分布式事务补偿——需要深度推理处理高复杂度场景 |
| T5 倒计时前端组件 | **MiniMax-M2.5** | 前端组件 + NTP 时间同步逻辑，质量中上 + 速度快（~60s），性价比最优 |
| T6 WebSocket 推送 | **qwen3-coder-plus** | 相对标准的 WebSocket 广播模式，速度最快（~40s），深度需求不高 |
| T7 监控告警面板 | **glm-5-turbo** | Prometheus 指标定义 + Grafana JSON 模板，结构化输出任务，速度快（~60s）且均衡 |

## 4. 关键风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| **T2 Lua 脚本竞态条件** — Redis 主从切换时 Lua 原子性可能失效 | 高 | 需在 T2 中设计 fallback 方案：Redis Cluster 模式 + 库存最终一致性校验 |
| **T4 消息丢失** — MQ 宕机导致订单丢失 | 高 | 生产者确认机制 + 本地消息表 + 消费者幂等，在 code review 阶段重点审查 |
| **T1-T2 接口契约不一致** — Group 1 依赖 T1 输出 | 中 | T1 完成后先输出 `types/seckill.d.ts` 类型定义文件，T2 严格按契约开发 |
| **T3 验证码服务选型** — 未指定具体 CAPTCHA 方案 | 中 | 定义 `CaptchaProvider` 接口抽象层，默认实现 Google reCAPTCHA，可插拔替换 |
| **T6-T4 时序耦合** — 推送依赖订单结果 | 低 | 定义 `OrderResultEvent` 事件接口，T4 产出事件、T6 消费事件，解耦实现 |
| **全链路压测缺失** — 各模块独立开发但未端到端验证 | 高 | 规划 Phase 2 集成测试：JMeter/K6 压测脚本 + 热库存预热 + 降级预案 |

## 5. 预估总耗时

按关键路径计算（T3 并行不计入）：

| 阶段 | 任务 | 模型 | 耗时 |
|------|------|------|------|
| Group 0 | T1 | kimi-k2.5 | ~100s |
| Group 0 | T3 (并行) | qwen3.5-plus | ~140s |
| Group 1 | T2 | kimi-for-coding | ~130s |
| Group 1 | T5 (并行) | MiniMax-M2.5 | ~60s |
| Group 2 | T4 | kimi-for-coding | ~130s |
| Group 3 | T6 | qwen3-coder-plus | ~40s |
| Group 4 | T7 | glm-5-turbo | ~60s |

**关键路径串行耗时**: 100 + 130 + 130 + 40 + 60 = **~460s (~7.7 min)**

T3 (~140s) 与 Group 0 的 T1 并行，不影响关键路径。T5 (~60s) 在 T2 执行期间并行完成，也不影响。

**预估总耗时: ~8 分钟**（含所有并行组）

> 建议: T2 和 T4 作为核心高并发模块，完成后应触发 **cross-review**，由 kimi-for-coding 和 kimi-k2.5 交叉审查 Lua 脚本和消息队列的可靠性逻辑。
